## 1. What this expedition is — and the roadmap restructure it lands

The frame work is its **own full-arc 0.7 expedition** (ideation → … → documentary → tag — a camp expedition is
the whole journey, not a truncated ideation pass). It is **not** a 0.6.x point-feature; it's the next *epoch*.

**Roadmap restructure (this supersedes ROADMAP.md's tiering — reality taught):**
- **0.7 = the FRAME** (009 + 066/067). The old 0.7 ("UNDERSTAND — research the subsystems") was vague *because
  the frame wasn't settled*. The frame **is** that spine — once ratified, the subsystem work stops being
  "invent each from scratch" and becomes "apply the frame." So the frame doesn't *displace* the old 0.7; it's
  the prerequisite that makes it tractable. ADR-066 + ADR-067 already *started* this epoch on the 0.6.1 branch.

  release-granularity: one expedition emits incremental tags (0.6.1, 0.6.2, …). The capstone already wires
  0.6.2 + 0.6.4, so separate expeditions would only manufacture a cross-expedition integration seam.

## 3. The build spine (the heart of the plan)

1. **Build the stroma** (ADR-067 → the stroma-builder: full-AST attributed base graph + induced views +
   require-installed-rust-analyzer-composed resolution + edge-provenance tiers + the snapshot-vs-live keystone).
2. **Full known-good migration** — re-point **every** existing consumer (scan · fingerprint · audit · effector,
   *and the peripheral helpers*) onto the stroma, validated against the **existing test suite as the known-good
   baseline**. ONE cumulative-context wave, not incremental. Why this, and why *full*:
   - **Known-good baseline:** migrated code on the new substrate either reproduces known outputs (the stroma is
     *proven correct*) or diverges (a stroma bug, found against a right answer). New-organ-on-new-substrate has
     two unknowns and no baseline; migration collapses it to one.
   - **Fluency + API-shaping:** the stroma's real intricacies/needs surface only from use. Let the *first*
     consumer be known-good migration code (focus purely on stroma ergonomics), so its consumer API is
     ergonomic *before* the organs we care about arrive.
   - **Full, in one wave (not spine-only):** agent migrations share accumulated context — helper #15 carries
     #1–14's gotchas, so *later* migrations come out *more* bulletproof. And a **half-migrated state is two live
     representations of the codebase = the exact dual-source-of-truth drift antigen exists to catch.** Full
     migration keeps the dogfood honest.
   - One activity, three payoffs: **validate · learn · shape-the-API**.
3. **Build new on the proven, familiar stroma** — the 0.6.x organ *evolutions* + their frame co-develop pieces
   (§5). Note: scan/fingerprint *are* 0.6.4/0.6.5's substrate, so step 2 already lays their foundation.
4. **Capstone** — join the shipped organs.

## 4. Macros × stroma — they don't change; they get *located* in it

- **Macros are write-side (declaration-emitters); the stroma is read-side (representation).** `#[antigen]` /
  `#[presents]` / `#[antibody]` / `#[antigen_tolerance]` keep emitting the same source attributes; the full-AST
  attributed graph **represents `#[presents(X)]` as a property on that function's node.** Markers don't move or
  change shape — they become graph data.
- **What changes (not the macros):** (1) the stroma-builder gains an **antigen-attribute ingestion** step —
  recognizes antigen's own attributes (the existing `ANTIGEN_OWNED_ATTRS` set in `digest.rs`, the one a memory
  flags as incomplete) and attaches them to nodes with provenance; (2) scan/fingerprint/audit **query the
  stroma** instead of each re-walking syn.
- **The deeper boundary — settle EMPIRICALLY during migration, don't pre-decide:** the stroma likely becomes the
  **source-of-truth**, and the source `#[presents]` attribute becomes a **materialized view** of a stroma
  property (deterministically derived, `sync`-reversible — 009 Part 4). The macro still *parses* the attribute
  (hand-authored + materialized alike), so it survives untouched; it just stops being the primary store. Exactly
  where authored-in-source vs derived-from-stroma draws is the kind of thing the migration is the right place to
  learn. *(Verify against the real code — `scan-reader`, the macros, `ANTIGEN_OWNED_ATTRS` — before asserting
  the change-surface. Substrate over model.)*

## 6. The wave / team shape — full arc, generative-first

**dream → research → deconstruct → converge → ratify → build → survey → documentary → tag**, with research as a
*continuous, embedded, queryable capability*, not a discrete wave.

- **dream** — a real, multi-perspective generative wave (think-big-dreamer · value-finder · expansionist · scout
  · naturalist). 009 is *substrate the dreamers dream FROM*, not a ceiling. **Not skipped because 009 exists** —
  it runs generative over the full substrate (present + future), reaching things we build 10 versions out. Its
  output is **dual**: material for *this* expedition's ADRs *and* deepened/new charters for the future (the
  ROADMAP's dreamer-protocol — deepen-don't-re-dream).
- **research** — embedded with the dreamers/deconstructors (real-time grounding: lattice/order theory, refinement
  types / design-by-contract, the immunology danger-model (Matzinger Signal-1/2), Ashby's requisite variety,
  category-theoretic framings) **+ a targeted question-driven block AFTER deconstruct** (answer the exposed open
  Qs). "There's always more research" is the default, not a hedge.
- **deconstruct** — aristotle + adversarial + embedded researchers; chews 009's kernel/axes + the §7 open Qs.
- **converge** — synthesize + emit the **handoff** ("here are the ratified ADRs; here's which downstream
  build-dreams re-dream against the new frame").
- **ratify** — the intent-layer ADR · the unified-corpus ADR · the coordinate-frame posture (the *shippable
  fruit* of the understand-work).
- **build** — the spine (§3). **survey/documentary/tag** — the full arc closes.
